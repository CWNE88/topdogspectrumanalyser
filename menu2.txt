──────────────────────────────────────────────────────────────────
1. THE SPAN PROBLEM — AND HOW menu2.txt ALREADY SOLVES IT
──────────────────────────────────────────────────────────────────
The current code's [SPAN] menu is source-dependent: sweep sources get
Span + Full Span; sample sources get only "Sample Rate". That exposes
a hardware implementation detail where a user expects a bandwidth
control. It's the main friction point.

The menu2.txt design moves Sample Rate into [INPUT] and gives [SPAN]
a clean universal surface (Span + Full Span + Zoom). This is correct.
A user never needs to think about sample rate from the Span button.

The one thing this requires in code that doesn't exist yet:
  When a user enters a span value on a sample source, the app must
  automatically select the nearest valid sample rate, not just update
  the frequency display object.

  RTL Samples valid rates:  250k, 1.024M, 1.44M, 1.8M, 2.0M, 2.048M, 2.4M
  HackRF Samples:           2M, 4M, 8M, 10M, 16M, 20M
  Microphone:               8k, 11.025k, 16k, 22.05k, 44.1k, 48k, 96k

  Snap rule: pick the smallest rate ≥ requested span. If requested
  span exceeds max rate, clamp to max and show a status message.
  Show "Span → 2.048 MHz (2.048 MS/s)" so the user knows the snap.

Full Span on sample sources = select maximum sample rate, keep centre.
Currently Full Span only works for sweep sources; this needs extending.

No Auto/Manual mode toggle is needed. "Span" entry IS auto by
definition. "Sample Rate" in INPUT gives the manual path for experts.

──────────────────────────────────────────────────────────────────
2. BROADER REDESIGN ASSESSMENT
──────────────────────────────────────────────────────────────────
The menu2.txt redesign is a more ambitious structural change. Overall
assessment and notes per section:

[SPAN] — Span / Full Span / Zoom
  Good. Zoom In / Zoom Out is a nice quality-of-life addition.
  Suggest keeping Zoom under a sub-button so top-level stays tight,
  which this design already does. No issues.

[INPUT] — Source / Mode / Sample Rate / Bandwidth / Source Options
  This is a significant conceptual improvement over the current design
  where source-switching is buried several levels deep in [INPUT 1].
  Grouping Source + Mode + Sample Rate together is logical.

  "Bandwidth > Auto / Manual" is ambiguous — this appears to overlap
  with the Span concept. Clarify: does Bandwidth here mean IF filter
  bandwidth (i.e. hardware bandwidth/sample rate), or RBW (resolution
  bandwidth)? If it's sample rate under another name, remove it — it's
  already handled in Sample Rate above. If it's a distinct concept,
  it needs a clearer label.

  Source Options → Microphone channel is good. RTL/HackRF gain control
  marked as future is sensible.

[ANALYSIS] — new button for FFT/PSD/Constellation/Duty Cycle
  This is a clear improvement. Currently these are buried inside
  [INPUT 1] → [RTL Samples] → FFT / PSD / Constellation, which is
  confusing — why is an analysis mode inside an input menu?
  Moving them to a dedicated [ANALYSIS] button is logical.

  Note: this means adding/relabelling a physical front-panel button.
  In the current layout, the "Input 1" button would become "Analysis"
  or a new button would be needed. Worth confirming whether the UI
  layout can accommodate this or if Input gets repurposed.

  FFT Settings sub-section (Window / Size) is clean. Currently Window
  and Sample Size are separate sub-items within FFT, which is the same
  depth — no regression there.

[MARKER] — cleaner grouping
  Grouping Frequency/Power readout under a Readout sub-item adds a
  level compared to the current flat list. Not sure that's worth it —
  4 marker buttons at top level (F1/F2/P1/P2) is already scannable.
  Could simplify: keep Mkr 1 Freq / Mkr 2 Freq / Mkr 1 Power /
  Mkr 2 Power at top level with the function buttons, as currently.

[TRACE] — consolidates Averaging, Hold, Persistence
  Current design splits: Averaging is in [DISPLAY], Hold is in
  [DISPLAY], Persistence is in [DISPLAY], Trace A/B is in [TRACE].
  Putting them all under [TRACE] is more logical — these are all
  data capture/comparison features, not display visual settings.
  This is a good reorganisation.

[DISPLAY] — slimmed down to purely visual settings
  Good. Display Line, Threshold, Excursion, Log Freq Axis all belong
  here. Removing Averaging/Hold/Persistence from here (moved to Trace)
  makes this cleaner.

3. SUMMARY: WHAT TO DO FIRST (in order of value / effort)
──────────────────────────────────────────────────────────────────
A. Fix SPAN for sample sources (medium effort, high value)
   Span entry auto-snaps to nearest sample rate. Full Span selects max.
   Sample Rate stays in INPUT as secondary control.
   This alone fixes the main UX complaint without any structural change.

B. Move FFT/PSD/Constellation/Duty Cycle to an Analysis menu
   (medium effort, high value — but may need button relabel on UI)
   Currently the deepest / most confusing part of the menu.

C. Move Averaging, Hold, Persistence from Display → Trace
   (low effort, good value — these belong with trace, not display)

D. [INPUT] restructure to Source / Mode / Sample Rate
   (higher effort — changes how source switching works in code)
   Only worth doing once A/B/C are done and if the current INPUT
   structure is still confusing.

E. Zoom in Span, Delta marker, future gain controls
   (lower priority — useful but not the main friction)

4. ONE THING TO WATCH: BUTTON IDENTITY
──────────────────────────────────────────────────────────────────
The current physical buttons map to specific keyboard shortcuts
(F, S, A, I, M, D, P). Any rename of a top-level button should
also update its keyboard shortcut and the CLAUDE.md / menu.txt
documentation. The menu2.txt redesign suggests N for Analysis,
which is a free letter — good choice.
